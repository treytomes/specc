Overview

What is this?

Specc takes a Markdown description of a program and lowers it through deterministic passes to a native .NET executable. An LLM does exactly two things: extract a structured spec from prose, and embed each graph node as a learned vector. Everything else is deterministic code.

CFG construction, stack lowering, IL emission, acceptance verification — all deterministic. The LLM is reserved for tasks that genuinely require semantic understanding: extracting a spec from prose, and producing comparable vector representations of graph nodes.

Core Idea

Programs are graphs. Graphs have nodes. Nodes have embeddings.

  1. Two programs that implement the same intent should be close in embedding space
  2. Prior compilations can inform future ones — the repository as a standard library
  3. Program structure becomes something that can be searched, compared, and eventually refined by gradient

"Not agentic. Not autocomplete. A compiler with a semantic layer."

Architecture

The Pipeline

00 MarkdownSpecPass — LLM extracts .spec from prose
05 StackIrPass — stack opcodes
01 ParseSpecPass — .spec → RawSpec
06 MsilGenerationPass — IL assembly text
02 SemanticGraphPass — typed node/edge graph
07 AssemblyEmitPass — native executable
03 EmbeddingPass — mxbai-embed-large, one vector per node
08 AcceptanceVerificationPass — diff stdout vs expected
03b SemanticNormalizationPass — cosine similarity reclassification
09 RepositoryPersistPass — store verified compilation
03c RepositoryRetrievalPass — prior compilations as context
03a NodeMlpPass — per-kind MLP refines embeddings
04 CfgPass — deterministic structural lowering
Incremental by artifact: if a pass's output file already exists, the pass is skipped. Deleting an artifact forces that pass and all downstream passes to re-run.

Hardest Problem

The Extraction Cliff

The hardest problem wasn't the compiler. It was the LLM front-end.

Hypothesis
ministral-3b would fail on Collatz (unbounded while + division + conditional assigns)
Reality
It did fail — but the cliff was bridgeable

What broke

  • ✗Model placed print: before while: in spec output (structural bug)
  • ✗Classifier selected wrong construct family (loop: instead of while:)
  • ✗Two sequential true_assign: blocks collapsed to one (YAML key collision)
  • ✗Expected output section not parsed (bare lines, not fenced)
  • ✗Bad extractions persisted to the repository, poisoning future retrievals
Solutions

How We Crossed the Cliff

Each failure became permanent pipeline infrastructure.

Failure Fix
Wrong construct family ConsistencyMissing promoted from warning to retry trigger
Print before while SemanticGraphPass relocation at flush time
YAML key collision Halve + StepCounter examples teach two-true_assign: separately
Bare expected output ParseExpectedOutputBlock extended to non-fenced lines
Repository poisoning RepositoryPersistPass gated on AcceptancePassed == true
Cold-start failures Verified prior specs injected into extraction prompt (repository-as-standard-library)
Spec 35

Embedding Geometry

The premise: programs with the same intent should cluster in embedding space. Results from scripts/geometry.py:

  • FizzBuzz vs CountDown clearly separated — different algorithms, different structure — ✓
  • BubbleSort vs FizzBuzz clearly separated — unrelated algorithms — ✓
  • BubbleSort vs SelectionSort nearly identical — borderline vs cluster threshold 0.928 ✗

The third result is the correct behavior. Two near-identical sorting algorithms should be close in embedding space. The cluster separation test is too strict when algorithms are genuinely similar. The premise holds. Raw Ollama embeddings are intent-sensitive.

Open Questions

What Remains

Two open questions before "complete":

Spec 41 — Node MLP Training

NodeMlpPass runs. Training does not.

NodeMlpPass runs in the pipeline with randomly initialized weights. The forward pass exists. The training loop — contrastive loss over Contains/DependsOn edges — is not yet implemented. Until training runs and converges, we cannot claim the MLP improves clustering over raw embeddings.

Spec 21 — Extraction Ceiling

ministral-3b has a reliability ceiling

The current extraction path works with scaffolding. ministral-3b cannot reliably produce well-formed graph JSON directly — three attempts, all failed. Direct graph extraction requires a ≥7B model. The .spec text intermediate is a workaround, not a solution.

Status

What "Complete" Means

Four criteria. Three done.

# Criterion Status
1 Programs with same intent cluster in embedding space ✓ Done
2 Node MLP refinement improves clustering measurably ⧖ Forward pass only — training pending
3 Unbounded conditional loop compiles and runs ✓ Done (Collatz)
4 Extraction front-end is not the bottleneck ~ Scaffolded, not solved

Beyond "complete": self-hosting (Spec 20), WASM backend (Spec 43), user-defined intrinsics (Spec 42).

Future

The Longer Horizon

If the MLP training loop works:

→
Programs become points in a learned metric space
→
"Find me a program like this one" becomes a nearest-neighbour query
→
New programs could be synthesized by analogy: embed the intent, retrieve the closest prior graph, adapt by gradient toward the acceptance criteria
→
This bypasses the extraction front-end entirely — the LLM's role shrinks to one embedding call per node. Not authoring. Not deciding. Just measuring semantic distance.

"The compiler as a search problem in program space."
